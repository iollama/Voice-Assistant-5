# Import / Export a Device Profile

The main settings page (`/`) can back up and transfer your whole setup as a single `.zip`
**profile**. A profile bundles:

- your **persona** (the system-prompt text) — as `persona.md`,
- your **settings** (volume + voice + language) — as `settings.json`,
- your **emoji** animations (all seven moods) — as raw `.bin` frames + `manifest.json`.

This is handy for moving a configured personality from one device to another, or keeping a
backup before you experiment.

> **Secrets are never exported.** Your Wi-Fi password and OpenAI API key are *not* included in
> a profile zip.

## Exporting

1. Open the settings page (`http://voice-agent-XXYY.local/` or `http://192.168.4.1/` in AP mode).
2. In **Import / Export Profile** (in the *Your Settings* zone), click **Export profile**.
3. Your browser builds `persona.zip` and downloads it. The device streams the
   files; the zip is assembled in your browser. It lands in your browser's normal Downloads
   location.

## Browse (ready-made personas)

The fastest way to get a new personality is **Browse** — a built-in gallery of ready-made
personas hosted in the project's GitHub repo. Click **Browse** in the Import / Export Profile
card; the portal fetches the catalog and shows the personas grouped by category, each with a
thumbnail. Click one, confirm, and it's applied live (persona text + emoji). Volume/voice/language
are left unchanged unless that persona ships its own `settings.json`.

Browse needs the browser to be online (it reads the catalog and persona files from
`raw.githubusercontent.com`). The personas are stored as **loose files** in the repo
(`persona.md` + one image per emotion), not zips — the portal downloads them directly and
imports them through the same path as a file import.

> For maintainers/forkers: the gallery source is set by two constants near the top of
> `voice_agent_5/portal_js.h` — `REPO` (owner/repo) and `BRANCH`. Point them at your own
> fork/branch to test, then lock them back to the official repo before release. The catalog
> itself is `personas/index.json`, regenerated by `pscripts/generate-personas-index.ps1`
> (run automatically by `pscripts/pack.ps1`).

## Importing

In the same card you can also import **from a file** or **from a URL**:

- **Import from file** → pick a `persona.zip` (or any zip containing some of the
  pieces below).
- **Import from URL** → paste a direct link to a profile zip and click **Fetch & import**.
  The URL must be a **direct, CORS-enabled link** (e.g. a GitHub *raw* or *release-asset* URL)
  and your browser must be online and able to reach it. If the host doesn't allow
  cross-origin reads, the browser blocks it and you'll see a CORS error — re-host the file on
  a CORS-friendly URL.

Import **applies live, no reboot**: persona overwrites the current persona, volume/voice/language
take effect (voice and language on the next session), and emoji frames replace your customizations.

### Partial imports are fine

The importer applies **whatever the zip contains** and skips the rest:

- A zip with only `persona.md` updates just the persona; volume, voice and emoji are left
  untouched.
- A zip with only `settings.json` updates volume/voice/language only.
- Missing emoji simply keep their current frames.

Anything **present but not importable** is skipped individually (and listed in the status
line) rather than failing the whole import — for example a `voice` or `language` value this
firmware doesn't offer, a non-numeric volume, or a persona longer than the 12,288-character limit.
(An out-of-range volume *number* is clamped to 0–100 and applied, not skipped.)

## Zip layout

```
persona.zip
├── persona.md          # the system-prompt text (plain text)
├── settings.json       # { "version": 1, "volume": 50, "voice": "marin", "language": "auto" }
├── manifest.json       # { "neutral": {"count": 8}, ... all 7 emotions }
├── neutral_0.bin       # 45000 bytes, RGB565 little-endian, 150x150
├── neutral_1.bin
├── ...
└── thinking_7.bin
```

`settings.json` carries a `version` field so the format can grow over time; unknown keys are
ignored on import. The emoji `.bin` / `manifest.json` format is identical to an emoji pack —
see [emoji-customization.md](emoji-customization.md) for the frame-format details. The emoji
section of a profile may also use **source images/videos** (`<emotion>.gif/.png/.jpg/.mp4`)
instead of `.bin` frames; the browser converts them on import.

## Profile vs emoji pack

The **emoji page** (`/emojis`) has its own, narrower export/import that deals with *emoji
only*:

| | Main page — **profile** | Emoji page — **pack** |
|---|---|---|
| File | `persona.zip` | `emoji-pack.zip` |
| Persona / settings | included | **ignored** (even if present in an imported zip) |
| Emoji exported | **all 7** moods (custom *or* shipped default) | **only the moods you customized** |
| Import applies | persona + settings + emoji | emoji only |

So importing a full profile on the emoji page applies only its emoji; import it on the main
page to also apply the persona and settings.

## Requirements & limits

- **Browser must be online.** The page loads the zip (and GIF-decoder) libraries from a public
  CDN, so import/export does not work in fully offline AP-only mode — same constraint as the
  emoji page. Connect to a normal network first.
- Persona is capped at **12,288 characters** (longer personas are skipped).
- Emoji writes are **rejected during an active voice turn** (RECORDING / THINKING / SPEAKING);
  that emoji is reported as skipped — retry when the device is idle. Persona and settings still
  apply.
